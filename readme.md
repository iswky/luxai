# LuxAI - Система парсинга и анализа тендеров

Этот проект предназначен для автоматического сбора данных о тендерах с госзакупок, извлечения технических характеристик с помощью ИИ (DeepSeek API) и удобного отображения результатов через веб-интерфейс.

## 🚀 Быстрый запуск через Docker

Самый простой способ запустить всё — использовать Docker Compose. Все компоненты (веб-интерфейс, база данных и периодический парсинг) запускаются автоматически.

### 📋 Предварительные требования
- Docker и Docker Compose
- API ключ DeepSeek (для парсинга характеристик)

### 🛠 Запуск
1. Склонируйте репозиторий.
2. Создайте файл `.env` в корне проекта и добавьте ваш ключ и интервал парсинга:
   ```env
   DEEPSEEK_API_KEY=ваш_ключ_здесь
   PARSER_INTERVAL_SECONDS=3600
   ```
3. Выполните команду:
   ```bash
   docker compose up --build
   ```

После запуска:
- **Веб-интерфейс:** [http://localhost:8000](http://localhost:8000)
- **База данных (Postgres):** `localhost:5432` (user: `user`, password: `pass`, db: `appdb`)
- **Автоматический парсинг:** Запускается автоматически в контейнере `parser` с заданным интервалом.

## 🏗 Структура проекта

- `backend/` — Django проект с веб-интерфейсом.
- `parser/` — Скрипты для сбора данных с zakupki.gov.ru и парсинга PDF через DeepSeek API.
- `db/` — SQL скрипты для инициализации базы данных.

---

## ⚙️ Компоненты системы

### 1. Веб-интерфейс (Backend)
Отображает список тендеров, их детали и позволяет управлять процессом анализа.
Основные страницы:
- `/applications/` — Список всех найденных тендеров.
- `/files/` — Управление документами.
- `/equipment/` — Каталог оборудования.

### 2. Автоматический Парсер (Parser)
Контейнер `parser` работает постоянно и управляет процессом сбора данных.
- **Динамическая настройка:** Вы можете изменить `PARSER_INTERVAL_SECONDS` в файле `.env` прямо во время работы. Планировщик заметит изменение, **немедленно запустит парсинг** и сбросит таймер.
- **Ручной запуск:** Если нужно запустить парсинг немедленно вручную:
  ```bash
  docker exec -it parser python parser/main.py
  ```

### 3. База данных
Используется PostgreSQL 15. Схема данных автоматически создается при первом запуске контейнера с помощью `db/init_schema.sql`.

## 🛠 Ручной запуск (для разработки)

Если вы хотите запустить проект локально без Docker:

1. **База данных:**
   Запустите скрипт для настройки локальной БД:
   ```bash
   chmod +x ./setup_local_db.sh
   sudo ./setup_local_db.sh
   ```

2. **Зависимости:**
   ```bash
   pip install -r requirements.txt
   playwright install chromium
   ```

3. **Настройка окружения:**
   ```bash
   export DEEPSEEK_API_KEY=ваш_ключ
   ```

4. **Запуск Django:**
   ```bash
   python backend/manage.py runserver
   ```

5. **Запуск парсера:**
   ```bash
   python parser/main.py
   ```

## 📂 Подробное описание кодовой базы

Ниже представлено детальное описание всех директорий, файлов и функций проекта. Проект разделен на три основных компонента: веб-интерфейс (`backend/`), скрипты парсинга (`parser/`) и базу данных (`db/`).

### 🌐 Backend (Django Веб-интерфейс)
Директория `backend/` содержит веб-приложение на базе Django, которое предоставляет пользовательский интерфейс для просмотра и управления тендерами, документами и оборудованием.

#### `backend/config/`
Конфигурация проекта Django.
- **`settings.py`**: Основной файл настроек Django. Содержит конфигурацию базы данных, установленные приложения, пути к статическим файлам и файлам медиа (например, `TENDERS_FILES_DIR`), настройки безопасности.
- **`urls.py`**: Главный маршрутизатор (URL-конфигурация) проекта. Подключает маршруты из приложения `webui`.
- **`wsgi.py` / `asgi.py`**: Точки входа для WSGI/ASGI совместимых веб-серверов (например, Gunicorn, Uvicorn) для деплоя приложения.

#### `backend/manage.py`
- Стандартная утилита Django для управления проектом (запуск сервера разработки `runserver`, создание миграций, запуск консоли).

#### `backend/webui/`
Основное приложение Django. Не использует стандартную ORM Django для тендеров, вместо этого работает напрямую с базой данных через `db_repository.py`.
- **`apps.py`**: Конфигурация приложения `webui`.
- **`models.py`, `admin.py`**: Могут содержать базовые модели или настройки админки (если используются).
- **`urls.py`**: Маршруты приложения веб-интерфейса (главная страница, список заявок, файлы, каталог оборудования и т.д.).
- **`db.py`**:
  - `get_connection()`: Устанавливает прямое подключение к базе данных PostgreSQL (схема `r_luxai`) через `psycopg2`.
- **`db_repository.py`**: Слой доступа к данным (Repository). Содержит все SQL-запросы для веб-интерфейса:
  - `db_column_exists()`: Проверяет наличие столбца в БД (для обратной совместимости старых дампов).
  - `fetch_applications()`, `fetch_application_detail()`: Извлечение списка тендеров и их детальной информации (требования, позиции) из таблиц `tenders` и `tender_positions`.
  - `fetch_files()`, `fetch_document_file()`, `fetch_application_file()`: Работа с метаданными файлов.
  - `fetch_equipment()`, `fetch_shops_for_position()`, `fetch_selected_items()`: Извлечение информации о каталоге оборудования и магазинах для позиций.
  - Набор утилит для форматирования (`normalize_components`, `format_money`, `format_date`, `humanize_range` и т.д.).
- **`views.py`**: Контроллеры (представления) приложения:
  - `home()`: Главная страница.
  - `applications_page()`, `application_detail_page()`: Страница со списком всех тендеров и детальная страница конкретного тендера.
  - `files_page()`: Управление и просмотр сохраненных файлов тендеров.
  - `equipment_page()`: Страница со списком оборудования из БД.
  - `open_document_file()`, `open_application_file()`: Отдача файлов тендеров пользователю (через `FileResponse`).
  - `update_prompt()`, `select_shop()`, `contract_page()`: Управление запросами, выбор магазина и генерация контрактов.
  - Внутренние функции (`_documents_roots`, `_find_tender_file`, `_file_response_from_path`): Безопасный поиск и выдача файлов из файловой системы.
- **`templatetags/dict_filters.py`**:
  - `get_item()`: Пользовательский фильтр шаблонов Django для доступа к элементами словаря по ключу.

### 🧠 Parser (Система сбора и обработки)
Директория `parser/` содержит логику сбора данных с zakupki.gov.ru, загрузки документов, их фильтрации и извлечения характеристик через DeepSeek API.

#### `parser/main.py`
- Главный скрипт для обработки очереди тендеров.
  - `process_queue()`: Извлекает тендеры из `processing_queue`, загружает файлы, фильтрует их и отправляет на парсинг в LLM.
  - `main()`: Запускает процесс парсинга страниц zakupki.gov.ru и затем запускает обработку очереди.

#### `parser/scheduler.py`
- Автоматизирует запуск парсера с заданным интервалом.
  - `run_parser()`: Запускает `main.py` в виде отдельного процесса.
  - `get_interval()`: Читает интервал из файла `.env` (`PARSER_INTERVAL_SECONDS`).
  - `main()`: Бесконечный цикл, который следит за изменениями интервала в `.env` и запускает парсер.

#### `parser/database/db.py`
- Взаимодействие с базой данных PostgreSQL для парсера (запись результатов парсинга).
  - `save_tender_to_db()`: Сохраняет данные тендера и извлеченные через LLM позиции (оборудование) в таблицы `tenders` и `tender_positions`.
  - `is_tender_in_db()`, `delete_tender_from_db()`: Проверка существования и удаление тендера.
  - `deduplicate_tenders_in_db()`: Удаляет возможные дубликаты тендеров в БД.
  - `ensure_processing_queue_table()`: Создает таблицу `processing_queue`, если ее нет. Эта таблица используется для отслеживания состояний тендера: загрузка файлов -> фильтрация -> парсинг LLM.
  - `add_to_processing_queue()`, `get_all_from_processing_queue()`, `update_processing_queue_field()`, `remove_from_processing_queue()`: Управление очередью задач.
  - Утилиты приведения типов: `_extract_number_value()`, `_find_typed_columns()`, `_to_int()`, `_truncate()`.

#### `parser/llm/`
Модуль для извлечения информации из PDF с использованием ИИ.
- **`llm.py`**: Интеграция с DeepSeek API.
  - `parse_pdf_to_json()`: Читает PDF, отправляет текст в DeepSeek API вместе с промптами (из `prompts_templates/`) и получает структурированный JSON с техническими характеристиками (ОЗУ, процессор, объем памяти и т.д.). Результат кэшируется в виде `.json` файла рядом с `.pdf`.
  - `_call_deepseek()`: Низкоуровневый вызов API DeepSeek.
  - `_clean_and_parse_json()`: Очистка ответа от Markdown форматирования и парсинг в dict.
- **`pdf_extractor.py`**: Извлечение текста и таблиц из PDF перед отправкой в LLM.
  - `ContentBlock`, `PDFExtractor`: Классы для группировки текста и таблиц.
  - `extract_as_stream()`, `blocks_to_markdown()`, `_table_to_markdown()`: Преобразуют PDF в Markdown формат (с сохранением таблиц) для удобного понимания моделью.

#### `parser/processor/`
Модуль обработки файлов.
- **`downloader.py`**: Скачивание файлов тендера.
  - `download_tenders_files()`: Скачивает все прикрепленные файлы к тендеру.
  - `extract_file_links_44FZ()`, `extract_file_links_223FZ()`, `get_link_to_file_page()`: Парсинг страниц тендеров 44-ФЗ и 223-ФЗ для получения прямых ссылок на файлы.
  - `download_file()`, `unarchive_file()`: Загрузка и распаковка архивов (RAR, ZIP, 7z).
- **`file_filter.py`**: Фильтрация ненужных документов и конвертация в PDF.
  - `file_filter()`, `filter_single_tender_files()`: Удаляет лишние файлы (по ключевым словам из `file_filter_keywords.txt`) и оставляет только технические задания (ТЗ).
  - `convert_files_to_pdf()`, `convert_word_to_pdf()`, `convert_excel_to_pdf()`, `convert1file2pdf()`: Конвертирует `.doc`, `.docx`, `.xls` в `.pdf` с помощью LibreOffice.
  - `check_keywords_in_file()`, `find_files_with_keywords()`: Ищет ключевые слова внутри документов (в тексте), чтобы определить, является ли файл техническим заданием.
- **`validator.py`**: Валидация тендеров.
  - `tender_validator()`, `_wipe_tender_files()`: Проверяет и удаляет неполные или ошибочные данные тендеров, очищает связанные с ними файлы.

#### `parser/scraper/`
Парсинг сайта Госзакупок.
- **`zakupki_scraper.py`**: Парсер zakupki.gov.ru.
  - Класс `TenderParser`: Инкапсулирует логику поиска.
  - `Parse_gos_zakupki()`: Главный метод. Итерирует по страницам выдачи.
  - `parse_page()`, `extract_tenders()`, `parse_all_pages()`: Извлекает информацию о тендерах (номер, цена, заказчик, ФЗ) со страницы выдачи. Для каждого найденного тендера вызывает `page_callback` для немедленной обработки, добавляет в БД (`add_to_processing_queue`).
  - `load_filters_from_file()`: Читает параметры поиска (например, регионы, ключевые слова) из `parser/params.txt`.
- **`city_scraper.py`**: Модуль для определения города по адресу.
  - `city_parse()`, `extract_city()`, `extract_city_from_address()`: Пытается нормализовать и извлечь название города из текстового адреса заказчика.

### 🗄 База Данных (`db/`)
SQL-скрипты и утилиты для БД PostgreSQL.
- **`init_schema.sql`**: Полный скрипт инициализации схемы базы данных `r_luxai`. Создает таблицы `tenders`, `tender_positions`, `processing_queue` и другие, а также устанавливает связи. Этот скрипт выполняется автоматически при первом старте контейнера PostgreSQL.
- **`db_dump.sql`, `dump-luxai-*.sql`**: Дампы базы данных для бэкапа и восстановления.
- **`luxai.py`**: Вспомогательный скрипт (если необходимо) для специфичных задач базы данных.

### 📄 Прочие файлы и шаблоны
- **`docker-compose.yml`**: Файл конфигурации Docker Compose. Запускает и связывает сервисы: `db` (Postgres), `backend` (Django) и `parser` (Python скрипт). Настраивает тома (volumes) для персистентного хранения базы и скачанных файлов.
- **`backend/Dockerfile`**: Описание образа для веб-интерфейса. Устанавливает Python-зависимости, Playwright (и Chromium) и LibreOffice для конвертации файлов.
- **`requirements.txt`**: Список Python зависимостей (Django, psycopg2, requests, bs4, pdfplumber, openai, и т.д.).
- **`prompts_templates/`**: Файлы (`prompt_comp`, `prompt_items1`, `prompt_items2`), содержащие текстовые шаблоны для DeepSeek API. Они описывают ИИ, как именно извлекать данные и в каком JSON-формате их возвращать.
